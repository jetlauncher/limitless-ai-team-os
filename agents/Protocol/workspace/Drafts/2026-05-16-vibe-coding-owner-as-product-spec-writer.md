# Main Issue

## Opening insight

Vibe Coding ไม่ได้แปลว่าเจ้าของธุรกิจทุกคนต้องกลายเป็น developer

มันแปลว่าเจ้าของธุรกิจต้องกลายเป็นคนเขียนโจทย์ให้ชัดขึ้น

นี่คือ shift ที่ใหญ่กว่าเรื่อง coding

เมื่อก่อน ถ้าคุณอยากสร้างระบบเล็ก ๆ ในบริษัท คุณต้องมี developer ต้องมี budget ต้องมี timeline ต้องมี requirement document ที่ทีม business กับทีม tech แปลกันไปแปลกันมา

วันนี้ AI ทำให้ระยะจาก “ไอเดีย” ไปถึง “prototype ที่จับต้องได้” สั้นลงมาก

แต่มีเงื่อนไขหนึ่งข้อ

โจทย์ต้องชัด

ถ้าโจทย์คุณมั่ว AI จะสร้างของมั่วให้เร็วมาก

ถ้าโจทย์คุณชัด AI จะช่วยเปลี่ยนประสบการณ์ของคุณให้กลายเป็นเครื่องมือได้

## The shift

จาก transcript มีตัวอย่างที่น่าสนใจมาก

มีคนสร้างระบบ e-commerce ของตัวเองโดยไม่ได้เขียน code เอง

ระบบนั้นไม่ได้แค่โชว์สินค้า

แต่มี:

- รายการคอร์ส
- วันที่ของแต่ละคอร์ส
- จำนวนที่นั่งคงเหลือ
- cart
- เลือกวันชำระเงิน
- เลือกจ่ายแบบบุคคลธรรมดาหรือนิติบุคคล
- หักภาษี 3% อัตโนมัติสำหรับนิติบุคคล
- upload slip
- ส่ง email confirmation ให้ลูกค้า

ใช้เวลาประมาณ 1 วันในการขึ้นระบบแรก และอีกไม่กี่วันในการปรับโจทย์

นี่คือสิ่งที่เมื่อก่อนต้องรอทีม tech หรือ agency

แต่วันนี้ operator ที่เข้าใจปัญหาจริงสามารถเริ่ม prototype เองได้

คำสำคัญคือ prototype

ไม่ใช่ enterprise system ตั้งแต่วันแรก

อย่าเริ่มจาก “เราจะสร้างระบบใหญ่ทั้งบริษัท”

เริ่มจาก mini-app ที่แก้ปัญหาเล็ก แต่ใกล้เงินจริง

## What it means for Thai business owners

ในธุรกิจไทย ปัญหาหลายอย่างไม่ได้ต้องการ software ใหญ่

มันต้องการเครื่องมือเล็ก ๆ ที่ทำให้ทีมทำงานเป็นมาตรฐานขึ้น

เช่น:

- ฟอร์มรับ lead ที่บังคับเก็บข้อมูลให้ครบ
- ระบบคำนวณราคาเบื้องต้น
- mini CRM สำหรับทีม sales เล็ก ๆ
- dashboard งาน follow-up ลูกค้า
- internal tool เช็ค stock / booking / seat availability
- quote generator สำหรับทีมขาย
- content brief generator จาก transcript
- complaint tracker จาก LINE chat

เมื่อก่อนงานพวกนี้มักถูกปล่อยไว้ใน Excel, LINE, Google Sheet, หรือหัวของคนเก่งคนเดียว

เพราะมัน “ยังไม่ใหญ่พอ” ที่จะจ้างทำระบบ

แต่จริง ๆ แล้วงานเล็กพวกนี้คือจุดที่เงินรั่ว

ลูกค้ารอนาน
ข้อมูลตกหล่น
ทีมตอบไม่เหมือนกัน
เจ้าของต้อง approve ทุกอย่าง

Vibe Coding ทำให้เจ้าของธุรกิจมีทางเลือกใหม่:

สร้าง version แรกให้เห็นก่อน

แล้วค่อยตัดสินใจว่าจะต่อยอดเป็นระบบจริง จ้างทีมทำให้แข็งแรง หรือทิ้งไปเพราะไม่คุ้ม

นี่คือวิธีคิดของ operator

ทดลองให้เร็ว แต่ไม่หลอกตัวเองว่า prototype คือระบบ production

## The operator move

สัปดาห์นี้ทำ **Mini-App Opportunity Map**

เปิด document แล้วตอบ 5 คำถามนี้

**1. งานอะไรที่ทีมทำซ้ำทุกสัปดาห์ แต่ยังอยู่ใน LINE หรือ Excel?**

เลือกมา 10 งาน

**2. งานไหนใกล้เงินที่สุด?**

เช่น lead, proposal, payment, booking, follow-up, customer issue

**3. งานไหนมี rule ชัดพอ?**

ถ้างานมี if/then ชัด AI และ mini-app จะช่วยได้ดี

เช่น ถ้าลูกค้าเป็นบริษัท ให้หัก 3% ถ้าเป็นบุคคลธรรมดา ไม่ต้องหัก

**4. งานไหนถ้าลด error ได้ 50% จะรู้สึกทันที?**

อย่าเลือกงานที่ดูเท่

เลือกงานที่ถ้าหายมั่วแล้วทีมเบาลงจริง

**5. version แรกต้องมีแค่ 3 feature อะไร?**

ตัดให้เหลือ 3 อย่าง

ถ้าเกินนี้ คุณกำลังสร้างระบบใหญ่ ไม่ใช่ mini-app

## Tool / workflow / agent example

ตัวอย่าง prompt สำหรับเริ่ม Vibe Coding:

```text
ผมเป็นเจ้าของธุรกิจ [ประเภทธุรกิจ]
ผมอยากสร้าง mini-app สำหรับ [งานซ้ำ]
ตอนนี้ workflow ปัจจุบันคือ:
1. ...
2. ...
3. ...

ปัญหาที่เกิดบ่อยคือ:
- ...
- ...

ช่วยถามคำถามผม 15 ข้อเพื่อทำ product spec ให้ชัดก่อนเขียน app
หลังจากผมตอบแล้ว ช่วยสรุปเป็น:
- user flow
- data fields
- core features 3 อย่าง
- approval points
- first prototype prompt สำหรับใช้ใน Google AI Studio / Lovable / Replit
```

นี่คือ Reverse Prompting ที่ควรใช้ก่อนสั่ง AI สร้าง app

อย่าเริ่มจาก “ช่วยทำ app ให้หน่อย”

ให้ AI interview คุณก่อน

เพราะคุณมี context ของธุรกิจ

AI มีความสามารถในการสร้าง

สองอย่างนี้ต้องเจอกันตรงกลาง

หลักการที่ดีคือ 10-80-10

- 10% แรก: คุณให้โจทย์, context, business rule, standard
- 80% กลาง: AI ช่วยสร้าง prototype, UI, code, structure
- 10% สุดท้าย: คุณตรวจ, test, ปรับ, และตัดสินใจว่าใช้จริงได้ไหม

อย่าข้าม 10% แรก

และอย่า outsource 10% สุดท้ายให้ AI ทั้งหมด

## Closing line

ในยุค AI สิ่งที่แพงขึ้นไม่ใช่ code

สิ่งที่แพงขึ้นคือความชัดของเจ้าของธุรกิจ
