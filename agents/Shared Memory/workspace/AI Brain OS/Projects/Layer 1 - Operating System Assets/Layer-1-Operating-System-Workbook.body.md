# Start Here — The AI Operating System

## The shift
Most companies do not need more AI tools. They need an operating system.

A tool answers a prompt. An operating system turns company knowledge into repeatable output.

## Simple definition
An AI Operating System combines:
1. **Knowledge** — what the company knows, believes, sells, promises, and has learned from real work
2. **Instructions** — role briefs, rules, workflows, approval gates, quality standards
3. **Outputs** — reports, drafts, follow-ups, content, dashboards, proposals, checklists
4. **Review loop** — human judgment that corrects the system and improves the next run

## Member outcome
By the end of Layer 1, a member should have:
- One company AI strategy map
- One Knowledge Loop checklist
- One first AI employee role brief
- One working 10-80-10 delegation template
- One department prompt pack they can use immediately
- One approval boundary map
- One 7-day implementation plan

## Operating principle
Do not start by asking, "Which AI tool should we use?"
Start by asking:
1. What work repeats every week?
2. What knowledge does this work require?
3. What output should AI produce?
4. What does a human still need to approve?
5. How will the result improve the system next time?

# 10-80-10 Rule — Human × AI Delegation

## Core idea
A useful way to think about AI work:
- **First 10%: Human direction**
- **Middle 80%: AI execution**
- **Final 10%: Human review**

The leverage is in the middle 80%. The quality comes from the human 10% on both sides.

## First 10% — Human direction
The human defines goal, context, standard, constraints, examples, approval boundaries, and definition of done.

## Middle 80% — AI execution
AI handles drafting, research, summarizing, comparing, formatting, preparing options, and turning messy input into structured output.

## Final 10% — Human review
The human checks accuracy, judgment, taste, customer risk, brand fit, strategic direction, and approval/rejection.

## Common failure modes
| Failure | What happens | Fix |
|---|---|---|
| Skip first 10% | AI guesses the task | Use KOI or CLEAR before delegation |
| Overload middle 80% | AI tries to do strategy + execution + approval | Split into draft mode first |
| Skip final 10% | Risky or off-brand output goes live | Add review owner and approval gate |
| No feedback loop | Same errors repeat | Store corrections in Knowledge Loop |

## Delegation template
```markdown
# 10-80-10 Delegation Brief

## First 10% — Human Direction
Goal:
Audience:
Context/source material:
Output format:
Quality standard:
Constraints:
Approval rules:

## Middle 80% — AI Execution
Tasks AI should perform:
1.
2.
3.

Options to prepare:
- Option A:
- Option B:
- Option C:

## Final 10% — Human Review
Review owner:
Review checklist:
- [ ] Accurate
- [ ] Useful
- [ ] On brand
- [ ] Safe to share
- [ ] Has clear next action

Decision:
- [ ] Approve
- [ ] Revise
- [ ] Reject
- [ ] Save as reusable template
```

# CLEAR Framework — Prompt / Request Builder v0

> Workspace note: I found references to CLEAR as a founder framework, but not a final canonical acronym/spec. This is a practical v0 for member-portal use and should be reviewed before becoming official doctrine.

## CLEAR = Context, Limits, Examples, Action, Review
CLEAR turns a rough request into a usable work order.

## C — Context
Give the business situation: audience, goal, prior context, source material, and business result.

## L — Limits
Set constraints: approvals, tone, length, sensitive data, assumptions not allowed.

## E — Examples
Show what good looks like: past winning post, good email, best proposal, brand voice sample, preferred report, bad example to avoid.

## A — Action
Ask for a specific output: draft, summarize, compare, score, rewrite, analyze, checklist, or plan.

## R — Review
Tell AI how the human will judge the result: accuracy, usefulness, brand fit, risk level, next-action clarity, decision-readiness.

## CLEAR template
```text
Context:
[Business situation, audience, source material, goal]

Limits:
[Constraints, approvals, sensitive data, tone, length]

Examples:
[What good looks like / what to avoid]

Action:
[The exact output to produce]

Review:
[How to score the result and what to flag]
```

## One-line version
"Given [context], within [limits], using [examples], produce [action/output], and self-review against [criteria]."

# Golden Triangle Strategy Map

## Core idea
Every useful AI workflow needs three things:
1. **Knowledge** — the business context that makes the answer yours
2. **Instructions** — the job description and rules for how work should be done
3. **Output** — a real artifact that lands inside the business

If one corner is missing, the system becomes a chatbot. If all three connect, AI becomes an operating system.

```text
              OUTPUT
       reports / drafts / actions
              /                    /                     /              KNOWLEDGE -------- INSTRUCTIONS
 company context       rules + role brief
```

## Strategy questions
### Knowledge
- What company knowledge does this workflow require?
- Where does that knowledge live today?
- Is it in people's heads, chat, files, CRM, calls, or Notion?
- Is it structured enough for AI to use?
- Who owns updating it?

### Instructions
- What role is the AI playing?
- What is it responsible for?
- What inputs should it use?
- What should it never do without approval?
- What does "good" look like?
- When should it escalate?

### Output
- What should the AI produce?
- Where should the output land?
- Who reviews it?
- What is the acceptance standard?
- What changes in the business if this output is produced every week?

## AI-fit scoring
| Criteria | Score 1 | Score 5 |
|---|---|---|
| Repetition | Rare / one-off | Happens weekly/daily |
| Knowledge availability | Mostly in someone's head | Clear docs, calls, CRM, examples |
| Output clarity | Vague | Clear artifact and format |
| Risk | High external/legal/money risk | Draft/internal/read-only first |
| Review owner | Nobody owns it | Clear human approver |
| ROI | Nice-to-have | Saves time or creates revenue |

Best first workflows: high repetition, clear output, low risk, obvious review owner.

## Department examples
| Department | Knowledge | Instructions | Output |
|---|---|---|---|
| Sales | Offers, objections, CRM notes | Follow-up style, approval rules | Draft follow-up + CRM summary |
| Marketing | Brand voice, past content, offers | Campaign objective, tone, examples | Content calendar + post drafts |
| Ops | SOPs, exception cases, checklists | Process steps, escalation rules | Daily ops report / checklist |
| HR | Role scorecard, culture, interview notes | Screening criteria | Candidate shortlist |
| Finance | Transactions, budgets, revenue targets | Reporting cadence, thresholds | Weekly finance summary |

# Knowledge Loop Primer

## Core definition
A Knowledge Loop is a system that makes company knowledge flow back into the business every time work happens.

It is not just SOPs, Google Drive, Notion folders, or a static knowledge base.

```text
Work happens → knowledge appears → capture → clean → connect → apply → correct → system improves
```

## 5C Framework
1. **Capture** — Pull knowledge out of heads, chats, calls, files, and real work.
2. **Clean / Structure** — Make knowledge readable by humans and AI.
3. **Connect / Retrieve** — Let AI retrieve the right knowledge when doing work.
4. **Copilot / Apply** — Use knowledge inside actual workflows.
5. **Correct / Improve** — Every wrong answer, new case, or better process updates the knowledge base.

## Key line
A knowledge base stores information. A Knowledge Loop makes information improve through use.

# KOI Method — Role Builder

## Definition
KOI = **Knowledge × Output × Instructions**.

It is the fastest way to turn a vague AI request into a usable AI employee brief.

Most people prompt AI like this:
> "Help me with sales."

KOI turns it into:
> "Use this company knowledge, produce this exact output, follow these instructions and approval rules."

## K — Knowledge
The context the AI needs to understand the business.
Examples: company overview, offer details, customer segments, past examples, sales objections, brand voice, SOPs, pricing rules, approval rules, sensitive data boundaries.

## O — Output
The artifact the AI must produce.
Examples: CRM-ready lead summary, 5-message follow-up sequence, weekly sales report, content calendar, candidate scorecard, customer reply draft, founder morning brief.

## I — Instructions
The role, process, standards, and boundaries.
Examples: role, approved sources, draft-only mode, escalation triggers, tone, steps, definition of done.

## KOI builder template
```markdown
# AI Employee Brief — [Role Name]

## K — Knowledge
Use these sources:
- Company:
- Offer:
- Customer:
- Past examples:
- Tools/data:
- Off-limits data:

## O — Output
Produce:
- Format:
- Length:
- Destination:
- Review owner:
- Due cadence:

## I — Instructions
Role:
Rules:
Tone:
Steps:
Approval gates:
Escalation triggers:
Definition of done:
```

## KOI quality check
- Is the knowledge specific enough?
- Is the output visible and reviewable?
- Are the instructions clear enough for a junior employee?
- Is there a human approval gate?
- Does the output land where the team already works?

# 7-Day First AI Employee Sprint

## Promise
In 7 days, your first private AI employee is set up with company memory, approved tools, and a weekly operating rhythm.

## Day 0 — Pre-qualification
Ask:
- What role do you wish you could hire this month?
- What task wastes the most time every week?
- Which tools does your team live in daily?
- What data is sensitive or off-limits?
- Who approves external emails, messages, payments, deletes, posts, or customer-facing outputs?

## Day 1 — Choose role, workflow, metric
Output: one AI employee role, one workflow, one success metric, approval gates.
Best first roles: Founder Executive Assistant, Sales Follow-Up Assistant, Research Assistant, Content Operations Assistant, Weekly Reporting Assistant.

## Day 2 — Build knowledge base
Collect company overview, offer/product details, customer profiles, past examples, SOPs/checklists, tone/brand examples, sensitive/off-limits data list.

## Day 3 — Build first workflow
Good first workflows: daily inbox/open-loop summary, weekly founder briefing, meeting notes to tasks, lead research to CRM draft, draft follow-up emails for approval, weekly sales/content report.

## Day 4 — Test in draft-only mode
Definition of done: client sees real output, approval rules are respected, no risky autonomous action, output is useful enough to review, agent has a weekly rhythm.

## Day 5 — Handoff demo
Show where the AI employee lives, what it can/cannot do, approvals, reports/logs, task requests, pause/offboarding.

## Day 7 — First useful report
Deliver first output/report, Loom walkthrough, what worked, what needs tuning, next workflow recommendation.

## Day 14 — ROI review
Ask: Did it save time? Did it catch anything useful? What felt annoying? What should we automate next? Is this worth expanding?

## One-line operating principle
Don't make the client think. Give defaults, ask only for decisions that affect business priority or safety, and deliver one useful output fast.

# Department AI Operating Rhythm

## Weekly rhythm
Each department should have one repeatable AI rhythm:
1. Input collection
2. AI draft / analysis
3. Human review
4. Approved output
5. Correction added back into Knowledge Loop

## CEO / Strategy
- Monday: weekly briefing and decision list
- Wednesday: opportunity / risk scan
- Friday: review open loops and next week priorities

## Sales
- Daily: lead follow-up draft queue
- Weekly: pipeline summary
- Monthly: objection pattern review

## Marketing
- Weekly: content batch from source library
- Daily: draft review / scheduling
- Monthly: performance learnings added to prompt library

## Operations
- Daily: exception report
- Weekly: SOP gap report
- Monthly: process improvement review

## HR
- Weekly: candidate screen and interview prep
- Monthly: onboarding FAQ update

## Finance
- Daily/weekly: cash/revenue snapshot
- Monthly: budget variance and decision memo

## Rule
If a workflow does not have a cadence, owner, and review rule, it is not an operating system yet.
